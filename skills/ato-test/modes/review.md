# レビューモード（出口ゲート）

既存のテスト成果物を受け取り、抜け漏れを検出して指摘するモード。前提は SKILL.md の「方針」「ID・トレーサビリティ規約」「共通規約」。

**このモードは「客先提出できるか」と「実施に移れるか」を判定する出口ゲート。**
チェックリストが go/no-go の合否表で、全項目を通れば提出・実施に進める。

## 手順

1. **対象を読む。** レビューするテスト成果物（および元の設計書・AC）を読む。
2. **チェックリストで突く。** 下のリストを順に当て、欠落・矛盾・規約違反を洗い出す。
3. **指摘を出す。** 各指摘は **「箇所（TC-ID/章）→ 何が欠けているか → どう直すか」** の順で具体的に、日本語で書く。
   重大度（必須修正／推奨／軽微）を付け、必須修正を先頭に並べる。
4. **go/no-go を判定する。** 件数サマリと、**客先提出・実施に進めるか（必須修正が残っていないか）** を最後に明言する。

## チェックリスト（＝提出・実施への合否表）

### A. AC カバレッジ（設計との整合）

- [ ] **網羅**：設計書に載っている全 AC に対応する TC があるか。AC があって TC がない仕様はないか。
- [ ] **紐づけ**：各 TC に AC-ID が記載されているか。`TC-xxx | AC-xxx` の形で書かれているか。
- [ ] **番号帯**：TC の番号帯が FUNC/AC の番号帯（ドメイン）と対応しているか。
- [ ] **リンク切れ**：参照している AC-ID が設計書に実在するか。削除・採番変更で孤立していないか。
- [ ] **仕様変更追従**：AC が変更・追加されたときに対応する TC も更新されているか。

### B. テストケースの品質

- [ ] **具体性**：手順と期待結果が入力値・期待値の形で書かれているか。「正しく」「適切に」等の曖昧語はないか。
- [ ] **粒度**：1 TC = 1 観点になっているか。複数の検証を1 TC に詰め込んでいないか。
- [ ] **状態網羅**：一覧/フォーム系で「正常 / 0件 / エラー / ローディング / 範囲外 / 権限なし / レスポンシブ」が漏れていないか。
- [ ] **非機能**：性能・SEO・アクセシビリティ・セキュリティの NFR に対応する TC があるか（対応する AC がある場合）。

### C. テストコードの整合（E2E・単体がある場合）

- [ ] **仕様書との一致**：コード内の TC-ID コメントが仕様書の TC テーブルと一致しているか。
- [ ] **アサーション**：`expect` / `assert` が書かれているか。テストが常にpassする（assert なし）コードはないか。
- [ ] **独立性**：テストが実行順序に依存していないか（前のテストの副作用を引き継がない）。
- [ ] **汎用性**：特定フレームワーク前提のコードがある場合、コメントで補足されているか。

### D. 客先体裁・版（仕様書）

- [ ] **章立て**：変更履歴・テスト環境・合格基準・残課題が揃っているか。
- [ ] **版・タグ**：提出済みなら版・Gitタグが入っているか。
- [ ] **残課題**：AC 未確定で TC を作れなかった箇所が「残課題」として明示されているか。

## 指摘の書き方（例）

> **[必須] TC-021（ニュース一覧0件）/ AC カバレッジ**
> AC-021「0件時に解除導線を表示する」に対応する TC がない。
> → test-spec.md の TC テーブルに TC-021 を追加し、E2Eテストの `news.spec.ts` にも対応する `test` を追記する。

> **[必須] TC-026 / アサーション欠落**
> E2Eテストの `test('TC-026 ...')` に `expect` が1つもない。テストが常にpassする状態。
> → 404 レスポンスまたはエラーページ表示の `expect` を追加する。

> **[推奨] TC-011 / 粒度**
> 「フォーム送信 → 完了表示 → メール送信」を1 TCで検証している。
> → 画面表示（TC-011a）とメール到達（TC-011b）を分割する。

## 総括の書き方（例）

> **判定：no-go（提出・実施に進めない）**
> 必須修正 2 件 / 推奨 1 件 / 軽微 0 件。
> AC-021 のTC未作成と TC-026 のアサーション欠落が必須。これらを潰せば go。
