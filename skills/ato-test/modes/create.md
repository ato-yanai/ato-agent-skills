# 作成モード

新規にテスト成果物を作る／既存に追記するモード。前提は SKILL.md の「方針」「ID・トレーサビリティ規約」「共通規約」。

## 着手前：入力ゲート

テストを始める前に、以下が揃っているかを確認する。**揃っていない場合は ato-design-doc へ戻し、AC を埋めてもらう。**

| 受け取るもの | 欠けると |
| --- | --- |
| 対象の設計書（FUNC/PAGE/COMMON が採番済み） | TC の番号帯・トレーサビリティが決まらない |
| 各仕様に AC が記載されている | テストケースが書けない（AC = テストの唯一の入力） |
| テスト環境（URL・ブラウザ・OS等） | test-spec.md の「テスト環境」が埋まらない |
| 合格基準（何をもって完了とするか） | go/no-go の判定ができない |

AC が空欄の仕様が1つでもある場合、**その仕様のテストは「作成不可（AC未確定）」として残課題に挙げる**。

## 種別の選び方

作成モードが選ばれたら、次を AskUserQuestion で確認する：

1. **何を作るか**（複数選択可）
   - テスト仕様書（`test-spec.md`）— AC → TC テーブル。客先提出の主文書
   - E2Eテスト（`*.spec.ts`）— Playwright 想定の自動テストコード
   - 単体テスト（`*.test.ts`）— Vitest 想定の単体テストコード
   - すべて（仕様書 + コード両方）

2. **対象スコープ**
   - どの FUNC/PAGE を対象とするか（全件か、特定ドメインか）

**テスト仕様書は常に最初に作る。** コードだけ先に作ることはしない（仕様書が TC の正本）。

## 手順

### 共通

1. **入力ゲートを確認する。** AC が揃っているか確認。不足は残課題化する。
2. **対象を決める。** 何を作るか・対象スコープを確定する。
3. **TCを採番する。** SKILL.md「ID・トレーサビリティ規約」に従い TC-xxx を振る。
   番号帯は FUNC の番号帯（AC が属するドメイン）に合わせる。

### テスト仕様書（test-spec.md）

4. `assets/test-spec.md` をテンプレとして読み、案件名・日付・テスト環境・合格基準を埋める。
5. 設計書から AC を読み取り、1 AC = 1 TC を原則に TC テーブルを埋める。
   - 複数 AC が密接に関連する場合は1 TC にまとめ、AC-ID を列挙する。
   - 設計書の「状態網羅」（0件・エラー・範囲外等）のACが漏れていないか確認する。
6. **テストコードがある場合**は「コード参照」列にファイルパス・TC-IDを記載する。

### E2Eテスト（*.spec.ts）

4. `assets/test-e2e.md` のテンプレを読む。
5. **仕様書の TC テーブルを基に** `describe` / `test` を構成する。1 TC = 1 `test`。
6. 各テストの先頭コメントに `// TC-xxx: テスト名 | AC-xxx` を必ず付ける。
7. 出力先は `tests/e2e/<ドメイン>.spec.ts` を提案（なければユーザーに確認）。

### 単体テスト（*.test.ts）

4. `assets/test-unit.md` のテンプレを読む。
5. **仕様書の TC テーブルを基に** `describe` / `it` を構成する。
6. 各テストの先頭コメントに `// TC-xxx: テスト名 | AC-xxx` を必ず付ける。
7. 出力先は `tests/unit/<対象モジュール>.test.ts` を提案（なければユーザーに確認）。

### 完了確認

8. 作成物と残課題（ACが揃っておらず TC を作れなかった箇所）を日本語で報告する。
9. **変更履歴は提出時のみ更新する**（提出モード参照）。日々の編集は Git へ。

## つまずきやすい点

- **AC が曖昧で TC にならない** → 入力値と期待値を具体化してから TC を書く。できなければ設計書へ差し戻す。
- **1 AC に複数の検証が混在している** → TC を分割する。1 TC = 1 観点。
- **E2E と単体の境界が分からない** → 画面操作・遷移・表示が絡む → E2E。ロジック・変換・バリデーション単体 → 単体テスト。
- **テスト環境が複数ある（PC/SP・ブラウザ違い）** → 仕様書の「テスト環境」に明記し、代表環境で実施する TC と全環境で実施する TC を区別する。
