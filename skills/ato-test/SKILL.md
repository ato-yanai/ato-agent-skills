# テスト（テスト仕様書・E2E・単体）

Webサイト/アプリの**客先提出テスト成果物**を、PMが違っても同じ品質・同じ書き方で作るためのスキル。
**ato-design-doc の AC を入力とし、TC 採番でトレーサビリティを取る**。

このスキルは**モード制**。起動したらまずモードを選び、選ばれたモードの手順ファイル（`modes/*.md`）に従う。

## 方針（これだけは守る）

- **結果・進捗・指摘の報告はすべて日本語で行う（絶対事項）。**
- **AC が入力。AC のない仕様はテストしない。** AC が空欄なら ato-design-doc へ戻してACを埋めてもらう。
- **全 TC に AC-ID を紐づける。** TC-ID を見れば何をテストしているかが分かる。TC は AC の具体化であり、独立した仕様を書かない。
- **テスト仕様書（Markdown）が主文書。** テストコード（.ts）は仕様書に紐づく付属成果物。
- **汎用**（フレームワーク非依存）。スタック固有の記述はコメントで補足する程度にとどめる。
- **客先提出を前提に版管理する。** 版・Gitタグは文書単位で進める（詳細は `modes/submit.md`）。

## 起動時：モードを選ぶ（必ず最初に行う）

**このスキルが起動したら、何より先に AskUserQuestion でモードを尋ねる。** 勝手にモードを決めない。

| モード | 用途 | 手順ファイル |
| --- | --- | --- |
| **① 作成** | 新規にテスト成果物を作る／追記する | `modes/create.md` |
| **② レビュー** | 既存テスト成果物の抜け漏れを検出して指摘する | `modes/review.md` |
| **③ 提出** | 客先提出に向け版を上げ、コミット＆タグ付けする | `modes/submit.md` |

## 構成

| 種別 | ファイル | 役割 |
| --- | --- | --- |
| モード手順 | `modes/create.md` | 作成モードの手順・種別の選び方 |
| モード手順 | `modes/review.md` | レビューのチェックリスト・指摘の書き方 |
| モード手順 | `modes/submit.md` | 提出フロー（版上げ・タグ付け） |
| テンプレ | `assets/test-spec.md` | テスト仕様書（主・客先提出文書） |
| テンプレ | `assets/test-e2e.md` | E2Eテストコードテンプレート（汎用Playwright） |
| テンプレ | `assets/test-unit.md` | 単体テストコードテンプレート（汎用Vitest） |

## ato-design-doc との関係

```
ato-design-doc（FUNC → AC を定義）
        ↓ AC-xxx を入力として受け取る
ato-test（AC → テスト成果物に変換）
    ├─ 仕様書モード → test-spec.md（TCテーブル）
    ├─ E2Eモード   → *.spec.ts（Playwright）
    └─ 単体モード  → *.test.ts（Vitest）
```

**ato-test は ato-design-doc の下流**。AC が確定していない仕様のテストは作らない。

## ID・トレーサビリティ規約

| 接頭辞 | 対象 | 採番単位 | 例 |
| --- | --- | --- | --- |
| `TC-` | テストケース | ドメイン番号帯（FUNC に対応） | `TC-010 ニュース一覧 正常表示` |
| `AC-` | 受け入れ基準（参照のみ） | ato-design-doc が採番 | `AC-021` |

- **TC は必ず AC に紐づく。** 1 TC : 1 AC が基本。複数 AC を束ねる場合は明示する。
- **番号帯は FUNC に合わせる。** ニュース(FUNC-0xx)のACならTC-0xx に。対応付けが明確になる。
- **TC-ID は採番後に変えない。** テストコード内のコメント（`// TC-010`）と仕様書が常に一致する。

## 成果物の種別と役割

| 種別 | テンプレ | 役割 | 客先提出 |
| --- | --- | --- | --- |
| **テスト仕様書** | `test-spec.md` | AC → TC テーブル。範囲・環境・合格基準を明記。主文書。 | 主 |
| **E2Eテスト** | `test-e2e.md` | ブラウザ操作を自動化（Playwright想定）。仕様書の付属。 | 付属 |
| **単体テスト** | `test-unit.md` | 関数・コンポーネント単位の検証（Vitest想定）。仕様書の付属。 | 付属 |

## 版管理（全モード共通）

- **テスト仕様書（test-spec.md）が版の基準。** E2E/単体テストの版は仕様書版に合わせて管理する。
- **版上げタイミング**：AC/仕様の変更に伴うTCの追加・修正・削除、または実施結果の更新で客先提出するとき。
- **タグ規約**：`test-spec-NN`（仕様書）/ `test-e2e-NN`（E2Eコード）/ `test-unit-NN`（単体コード）。
- **変更履歴テーブル**は客先提出した版だけを記録。日々の編集は Git へ。
- 提出フロー詳細は `modes/submit.md`。

## 共通規約

### 章立て標準

`# タイトル — 案件名` → `## 変更履歴` → 本文 → 必要なら `## 前提・制約` `## 用語集`。

### テストケースの書き方

- **手順と期待結果をセットで書く。** 「ページを開く → ○○が表示される」まで書いて1 TC。
- **曖昧語を使わない。** 「正しく」「適切に」はNG。入力値と期待値を具体的に書く。
- **1 TC で1つの観点を検証する。** 複数の検証を1 TC に詰め込まない。
- **状態網羅**：一覧/フォーム系で「正常 / 0件 / エラー / ローディング / 範囲外 / 権限なし / レスポンシブ」を必ず潰す。
