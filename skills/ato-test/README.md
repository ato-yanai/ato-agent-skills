# ato-test

Webサイト/アプリの**客先提出テスト成果物**を、担当者が違っても同じ品質・同じ書き方で作るためのスキルです。

> このファイルは**人間向けの概要**です。エージェントへの指示の正（Single Source of Truth）は
> [`SKILL.md`](SKILL.md)。モード別の手順は [`modes/`](modes/)、テンプレは [`assets/`](assets/) にあります。

---

## 何をするスキルか

[ato-design-doc](../ato-design-doc/) が定義した **受け入れ基準(AC) を入力**とし、テストケース(TC)採番で
トレーサビリティを取りながら、テスト仕様書・E2Eテスト・単体テストを生成します。

```
ato-design-doc（FUNC → AC を定義）
        ↓ AC を入力として受け取る
ato-test（AC → テスト成果物に変換）
    ├─ テスト仕様書（客先提出の主文書）
    ├─ E2Eテスト（Playwright 汎用）
    └─ 単体テスト（Vitest 汎用）
```

**AC のない仕様はテストしない。** AC が空欄なら ato-design-doc へ戻して埋めてもらいます。

---

## テスト成果物（構成）

| 種別 | テンプレ | 役割 | 客先提出 |
| --- | --- | --- | --- |
| テスト仕様書 | [`assets/test-spec.md`](assets/test-spec.md) | AC → TC テーブル。範囲・環境・合格基準を明記。主文書。 | 主 |
| E2Eテスト | [`assets/test-e2e.md`](assets/test-e2e.md) | ブラウザ操作の自動テスト（Playwright 想定・汎用） | 付属 |
| 単体テスト | [`assets/test-unit.md`](assets/test-unit.md) | 関数・コンポーネント単位の自動テスト（Vitest 想定・汎用） | 付属 |

> **IDの接頭辞:** `TC-`＝テストケース（本スキルが採番）。`AC-`＝受け入れ基準（ato-design-doc が採番、参照のみ）。
> TC の番号帯は FUNC の番号帯に対応させる（例：ニュース FUNC-0xx → TC-0xx）。

## 3つのモード

起動したらまず **AskUserQuestion でモードを選ぶ**（勝手に決めない）。

### ① 作成 — `modes/create.md`

テスト成果物を新規作成／追記するモード。

- **入力ゲート確認から始める。** AC が揃っていない仕様のテストは残課題として明示し、作らない。
- **テスト仕様書を最初に作る。** E2E・単体はその後。コードだけ先に作ることはしない（仕様書が TC の正本）。
- `assets/` のテンプレをコピーし、1 AC = 1 TC を原則に TC テーブルを埋める。
- 各テストコードに `// TC-xxx: テスト名 | AC-xxx` コメントを必ず付ける。

### ② レビュー — `modes/review.md`

既存テスト成果物の抜け漏れを検出して指摘するモード。**「客先提出・実施に進めるか」を判定する出口ゲート。**

- **A. ACカバレッジ / B. TCの品質 / C. コードの整合 / D. 客先体裁・版** の4区分でチェック。
- 各指摘は **「箇所（TC-ID/章）→ 何が欠けているか → どう直すか」** の順で、重大度付きで日本語で書く。
- 最後に **go/no-go を明言**する。

### ③ 提出 — `modes/submit.md`

客先提出時に版上げ・タグ付けを行うモード。

- **テスト仕様書（test-spec.md）が版の基準。** E2E・単体の版は仕様書版に合わせて管理。
- タグ規約: `test-spec-NN` / `test-e2e-NN` / `test-unit-NN`（NN は版番号の2桁）。
- タグ付けには **ato-design-doc の `scripts/tag-submission.mjs` を共用する**（汎用スクリプト）。
- push は自動で行わず、コマンドを促すだけ。

---

## 使い方

1. このスキルを起動する（テストの作成・レビュー・提出を依頼する）。
2. 提示される3モードから選ぶ。
3. 選んだモードの手順に従って進める。**結果・進捗・指摘の報告はすべて日本語**。
