# Git Commit Message

ステージされた変更から、[Conventional Commits](https://www.conventionalcommits.org/) 準拠で
**あとから読んで意味がわかる**コミットメッセージを生成・レビューするスキル。

## 3 つの原則（これだけは守る）

1. **1 コミット = 1 目的（アトミック）**。無関係な変更を混ぜない。混ざっていたら分割を提案する。
2. **subject は「何を」、body は「なぜ」**。差分を見ればわかる *how* ではなく、差分からはわからない *why* を残す。
3. **機械にも人にも読ませる**。`<type>(scope): 要約` の形を守れば、人が grep でき、ツールが changelog を自動生成できる。

## 禁止事項（絶対に守る）

- **`Co-Authored-By:` などの共同編集者トレーラーを付けない。** 「Claude」「Anthropic」を含む Co-Authored-By や、`🤖 Generated with ...` 等のツール署名・宣伝行をコミットメッセージに**一切入れない**。単独の作者として書く。
- **説明部分を英語で書かない。** subject の説明・body・フッターの文章はすべて**日本語**で書く（`<type>(scope)` ラベルや `BREAKING CHANGE:` / `Closes:` などの規約キーワードのみ英語のまま）。

## 手順

1. 変更内容を把握する。可能なら `git diff --staged`（なければ `git diff`）と `git status` を確認する。
2. 変更が複数の目的にまたがっていれば、**コミット分割を先に提案**する。
3. 主目的に合う **type** を選ぶ（下の「type の選び方」を参照）。
4. 影響範囲が **1 語**で言えるなら `type(scope):` を付ける（例 `fix(auth):`）。**scope の表記はプロジェクト内で統一**する（`auth` と `authentication` を混在させない）。1 語で言い切れないなら付けない。
5. **subject** を日本語で簡潔に要約する（type と scope は英語のまま）。
   - 末尾に句点（。）を付けない
   - 全角25／半角50文字以内を目安（70 を超えると `git log --oneline` で切れる）
   - 「〜する」「〜を追加」など、何をしたかが一目で分かる言い切りにする
6. 説明が要るなら空行を 1 行挟んで **body** を書く。**1 行 72 桁で折り返し**、差分からは分からない **why（なぜ）** と **what（何を変えたか）** を日本語で書く。
   - **why は散文で、変更点の列挙は箇条書き（`-`）で**書いてよい。複数の独立した変更を並べるなら箇条書きが読みやすい。
   - ただし箇条書きで *why* を 1 行ずつ潰さない（理由は散文、変更点リストは箇条書き、と使い分ける）。`*` より `-` を使い、継続行はインデントして 72 桁で折り返す。
7. **破壊的変更**は type/scope の直後に `!`、またはフッターに `BREAKING CHANGE: <説明>` を書く（例 `feat(api)!: 廃止予定の v1 を削除`）。MAJOR に相当するので、利用側のコード変更を要する変更は必ず明示する。
8. 関連 issue はフッターに `Refs: #123` / `Closes: #123`（フッターは `Key: value` 形式）。

## 構造

```
<type>[(scope)][!]: <description>   ← header（必須・1行）

[body]                              ← 任意・空行を1つ挟む・72桁で折り返す

[footer(s)]                         ← 任意・空行を1つ挟む
```

- header と body、body と footer の間はそれぞれ**空行 1 行**で区切る。
  この空行がないと `git log` / `shortlog` / `rebase` が subject と body を区別できない。

## type 一覧

| type | 用途 | SemVer 影響 |
| --- | --- | --- |
| `feat` | 新機能 | MINOR |
| `fix` | バグ修正 | PATCH |
| `docs` | ドキュメントのみ | なし |
| `style` | 空白・整形など挙動に影響しない変更 | なし |
| `refactor` | 挙動を変えないコード整理 | なし |
| `perf` | 性能改善 | PATCH |
| `test` | テストの追加・修正 | なし |
| `build` | ビルドシステム・依存関係 | なし |
| `ci` | CI 設定・スクリプト | なし |
| `chore` | その他の雑務 | なし |
| `revert` | 過去コミットの取り消し | 状況による |

## type の選び方（迷ったとき）

上から順に当てはまるものを選ぶ。複数当てはまるならコミット分割を疑う。

1. 利用者から見た**挙動が変わるか**？
   - バグを直した → `fix`
   - 新しいことができるようになった → `feat`
2. 挙動は変わらない。**何を触ったか**？
   - ドキュメントだけ → `docs`
   - テストだけ → `test`
   - CI 設定だけ → `ci` ／ ビルド・依存だけ → `build`
   - 整形・空白・セミコロンだけ → `style`
   - コード構造を整理（挙動不変） → `refactor`
   - 速くしただけ → `perf`
3. どれにも当てはまらない雑務だけ → `chore`（最後の手段）

## Examples

**バグ修正 + 理由（body あり）**

```
fix(auth): 失効トークンで500を返さないようにする

アクセストークンが失効している場合に500ではなく401を返す。
ペイロードをデコードする前に失効を確認するよう順序を入れ替えた。

Closes: #482
```

**ドキュメントのみ（subject だけ）**

```
docs: READMEにインストール手順を追記
```

**変更点を箇条書きで列挙（body あり）**

```
refactor(parser): トークナイザを独立モジュールに切り出す

字句解析がパーサ本体と密結合していて単体テストが書けなかったため、
責務を分離した。挙動は変えていない。

- tokenizer.ts を新設し字句解析ロジックを移動
- parser.ts はトークン列を受け取るだけに変更
- 既存の結合テストは緑のまま維持
```

**破壊的変更**

```
feat(api)!: 廃止予定の v1 エンドポイントを削除

BREAKING CHANGE: /v1/* は応答しなくなる。利用側は /v2/* へ移行すること。
```

**取り消し**

```
revert: "feat(search): あいまい検索を追加"

This reverts commit 676104e.
本番でレイテンシが悪化したため一旦戻す。
```

**その他の短い例**

```
feat(cart): クーポン併用を許可する
fix(api): 欠損リソースに500ではなく404を返す
refactor(parser): トークナイザを関数に切り出す
perf(list): 仮想スクロールで初期描画を約40%短縮
```

## Edge cases

- **複数の無関係な変更が 1 コミットに混在** → 分割を提案する（`git add -p` / `git restore --staged` で目的ごとに分ける）。
- **type が判断できない** → 変更ファイルの種類（テスト/設定/ソース/ドキュメント）から推定し、迷っても `chore` に逃げず最も具体的な type を選ぶ。
- **subject が 50 文字に収まらない** → 詳細を body に移し、subject は要約に留める。それでも長いならコミットが大きすぎる兆候。
- **WIP・後でまとめる前提の小コミット** → メッセージに凝らず、push 前に `git rebase` で squash して整える方針を伝える。
