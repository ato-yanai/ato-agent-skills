# 作成モード

新規に設計書を作る／既存に追記するモード。前提は SKILL.md の「方針」「共通規約」「設計の原則」。

## ドキュメント種別の選び方

新規 CMS 案件（microCMS + Next.js 等）のデフォルトは、まず **親を 2 本（機能一覧・サイト全体）** 立て、
そこから各設計書を埋める。

1. **機能一覧**（必須・最初）— 機能を出し切って採番。網羅とトレーサビリティの起点。
2. **サイト全体設計**（必須・最初）— サイトマップ／画面一覧(SC)／遷移／URL設計。SC を確定させる。
3. **microCMS設計書**（CMS案件なら必須）— コンテンツモデル。絞り込み/ソートに使うフィールドをここで確定。
4. **ページ設計書**（必須）— 画面単位。データソース・状態網羅・SEO・受け入れ基準。
5. **共通処理設計書**（横断機構があれば必須）— 絞り込み/ページャ/META/フォーム/エラー処理など。

案件条件で足すもの：既存サイトからの移行があれば**データ移行計画**、会員/管理機能があれば**権限・公開ワークフロー**を
microCMS設計書の章として厚くする。小規模 LP 等で横断機構が無ければ共通処理設計書は省略可
（ただし 404/500・フォーム送信があるなら最低限 CP を立てる）。

## 手順

1. **対象を決める。** 上の「種別の選び方」で作る設計書を選ぶ。新規案件なら機能一覧とサイト全体から着手する。
2. **テンプレを読んでコピーする。** 該当 `assets/*.md` をテンプレとして読み、案件名・日付・内容を埋める。
   出力先はプロジェクトの設計書置き場（無ければ `docs/design/<種別>.md` を提案）。
3. **採番して埋める。** SKILL.md「ID・トレーサビリティ規約」に従い FN/SC/CM/CP/AC を振る。**FN↔SC は多対多**で、
   関連は ID 列で紐付ける（同名行を作らない）。ドメインごとに番号帯を分ける。
4. **受け入れ基準(AC)を必ず書く。** 各仕様項目に AC を採番して併記する。「状態網羅の固定リスト」を必ず潰す。
5. **親へ反映する。** 新しい FN/画面は機能一覧・サイト全体に登録し、参照 ID（SC/CM/CP/AC）がすべて解決することを確認。
6. **microCMS は既存スキルに合わせる。** フィールド種別・型・取得条件・Webhook の表記を ato-microcms-* と統一する。
7. **変更履歴は提出時のみ。** 日々の編集は Git のコミットへ（変更履歴テーブルは提出版だけ／提出モード参照）。
   最後に作成物と残課題（AC 未記入箇所等）を日本語で報告する。

## Examples

**入力:** 「ニュース一覧ページ（絞り込み・ソート・ページネーション・ページ毎META）の設計書を作って」

**出力（どの設計書に何を書くか）:**
- 機能一覧：`FN-010 一覧表示 / FN-011 絞り込み / FN-013 ページネーション / FN-014 一覧META・ページNo付与` を追記。
  関連画面 `SC-003`、関連 API `CM-002 news`、関連 `CP-005/CP-006`、各 AC を採番。
- サイト全体：`SC-003` を画面一覧へ。URL設計 `/news?category=&sort=&page=N`、一覧→詳細の遷移。
- microCMS：`CM-002 news` のフィールド・絞り込み/ソート可フィールド・取得条件（filters/orders/limit/offset）。
- ページ設計書：`SC-003` の要素・状態網羅・絞り込み/ソート/ページャ UI・META 文言・レスポンシブ・AC。
- 共通処理：`CP-005 一覧フィルタ機構 / CP-006 ページネーション機構 / CP-007 一覧META生成（ページNo付与）`。

## つまずきやすい点

SKILL.md「設計の原則」を参照（同名機能・別ドメイン・行の爆発・META2か所・relation・AC が書けない場合）。
