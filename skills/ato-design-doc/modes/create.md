# 作成モード

新規に設計書を作る／既存に追記するモード。前提は SKILL.md の「方針」「設計の流れと前後関係」「共通規約」「設計の原則」。

## 着手前：入力ゲート（要件定義から受け取るもの）

設計を始める前に、要件定義側から次が揃っているかを確認する。**揃っていない項目は空欄のまま進めず、
「客先確認事項」として残課題に挙げる**（AC が書けないのは仕様未確定のサイン）。

| 受け取るもの | 起こす設計書 | 欠けると |
| --- | --- | --- |
| 機能リスト＋優先度 | ① 機能一覧(FUNC) | FUNC を採番できない／必須・任意が決まらない |
| 対象範囲（公開/管理/移行の有無） | ① 前提・制約、③の章立て | 移行計画・権限設計の要否を判断できない |
| 必要なページ／サイトマップ案 | ② サイト全体(PAGE) | PAGE が確定せずページ設計に入れない |
| URL方針・多言語/ドメイン要件 | ② URL設計 | クエリ規約・canonical を決められない |
| コンテンツ種別と各項目（誰が何を入力するか） | ③ microCMS(CMS/フィールド) | TS型・必須・絞り込み軸が確定しない |
| 更新運用・公開ワークフロー・権限 | ③ 権限/公開ワークフロー・Webhook | 下書き/予約公開/プレビュー仕様が書けない |
| デザイン（ワイヤー/カンプ） | ④ ページ設計（要素・レスポンシブ） | 要素表・SP/PC 挙動が埋まらない |
| 横断的な振る舞い（絞り込み軸・件数・フォーム・404方針） | ⑤ 共通処理(COMMON) | COMMON の仕様・AC が書けない |
| 非機能の目標値（性能/SEO/A11y/対応環境/セキュリティ） | ① NFR | NFR を宣言できず AC に落ちない |

理想の引き継ぎは「**機能リスト＋優先度／サイトマップ／コンテンツ項目＋運用／デザイン／非機能目標**」の5束。
これが揃えば①〜⑤を全部埋められる。揃わない分は埋めずに残課題化し、確認待ちを明示する。

## ドキュメント種別の選び方

新規 CMS 案件（microCMS + Next.js 等）のデフォルトは、まず **親を 2 本（機能一覧・サイト全体）** 立て、
そこから各設計書を埋める。

1. **機能一覧**（必須・最初）— 機能を出し切って採番。網羅とトレーサビリティの起点。
2. **サイト全体設計**（必須・最初）— サイトマップ／画面一覧(PAGE)／遷移／URL設計。PAGE を確定させる。
3. **microCMS設計書**（CMS案件なら必須）— コンテンツモデル。絞り込み/ソートに使うフィールドをここで確定。
4. **ページ設計書**（必須）— **1ページ＝1ファイル `page-<slug>.md`**。`assets/page.md` をコピーして画面ごとに作る。
   データソース・状態網羅・SEO・受け入れ基準。各ページが独立文書として版・タグを持つ（slug 規約は SKILL.md「ファイル分割と版の粒度」）。
5. **共通処理設計書**（横断機構があれば必須）— 絞り込み/ページャ/META/フォーム/エラー処理など。1ファイル（分割しない）。

案件条件で足すもの：既存サイトからの移行があれば**データ移行計画**、会員/管理機能があれば**権限・公開ワークフロー**を
microCMS設計書の章として厚くする。小規模 LP 等で横断機構が無ければ共通処理設計書は省略可
（ただし 404/500・フォーム送信があるなら最低限 COMMON を立てる）。

## 手順

1. **入力ゲートを確認する。** 上の「入力ゲート」で要件定義からの引き継ぎが揃っているか確認。
   不足は埋めずに「客先確認事項」として残課題化する（空欄のまま設計を進めない）。
2. **対象を決める。** 下の「種別の選び方」で作る設計書を選ぶ。新規案件なら機能一覧とサイト全体から着手する。
3. **テンプレを読んでコピーする。** 該当 `assets/*.md` をテンプレとして読み、案件名・日付・内容を埋める。
   出力先はプロジェクトの設計書置き場（無ければ `docs/design/<種別>.md` を提案）。
4. **採番して埋める。** SKILL.md「ID・トレーサビリティ規約」に従い FUNC/PAGE/CMS/COMMON/AC を振る
   （**接頭辞＋連番**。接頭辞でどの設計書かが分かる。`PAGE-` の実体ファイルは site.md の画面一覧で `page-<slug>.md` に対応づく）。
   **FUNC↔PAGE は多対多**で、関連は ID 列で紐付ける（同名行を作らない）。ドメインで番号帯を分ける。
5. **受け入れ基準(AC)を必ず書く。** 各仕様項目に AC を採番して併記する。「状態網羅の固定リスト」を必ず潰す。
6. **親へ反映する。** 新しい FUNC/画面は機能一覧・サイト全体に登録し、参照 ID（PAGE/CMS/COMMON/AC）がすべて解決することを確認。
7. **microCMS は既存スキルに合わせる。** フィールド種別・型・取得条件・Webhook の表記を ato-microcms-* と統一する。
8. **変更履歴は提出時のみ。** 日々の編集は Git のコミットへ（変更履歴テーブルは提出版だけ／提出モード参照）。
   最後に作成物と残課題（AC 未記入箇所等）を日本語で報告する。

## Examples

**入力:** 「ニュース一覧ページ（絞り込み・ソート・ページネーション・ページ毎META）の設計書を作って」

**出力（どの設計書に何を書くか）:**
- 機能一覧：`FUNC-010 一覧表示 / FUNC-011 絞り込み / FUNC-013 ページネーション / FUNC-014 一覧META` を追記。
  関連画面 `PAGE-003`、関連 API `CMS-002 news`、関連 `COMMON-005/COMMON-006`、各 AC を採番。
- サイト全体：`PAGE-003` を画面一覧へ。URL設計 `/news?category=&sort=&page=N`、一覧→詳細の遷移。
- microCMS：`CMS-002 news` のフィールド・制約・絞り込み/ソート可フィールド（具体的な取得条件はページ側）。
- ページ設計書：`page-news.md`（PAGE-003）に要素・状態網羅・取得条件（filters/orders/limit/offset）・絞り込み/ソート/ページャ UI・META 文言・レスポンシブ・AC。
  site.md の画面一覧「詳細ファイル」列を `page-news.md` に。詳細ページは別ファイル `page-news-detail.md`（PAGE-004）。
- 共通処理：`COMMON-005 一覧フィルタ機構 / COMMON-006 ページネーション機構 / COMMON-007 一覧META生成（ページNo付与）`。

## つまずきやすい点

SKILL.md「設計の原則」を参照（同名機能・別ドメイン・行の爆発・META2か所・relation・AC が書けない場合）。
