# レビュー・監査モード

既存の設計書を受け取り、抜け漏れを検出して指摘するモード。前提は SKILL.md の「方針」「共通規約」「設計の原則」。
属人化した「見落とし」を構造的に潰すのが目的。

## 手順

1. **対象を読む。** レビューする設計書（および関連する親＝機能一覧・サイト全体）を読む。
2. **チェックリストで突く。** 下のリストを順に当て、欠落・矛盾・規約違反を洗い出す。
3. **指摘を出す。** 各指摘は **「箇所（ID/章）→ 何が欠けているか → どう直すか」** の順で具体的に、日本語で書く。
   重大度（必須修正／推奨／軽微）を付け、必須修正を先頭に並べる。
4. **総括する。** 件数サマリと、特に提出前に潰すべき項目を最後にまとめる。修正まで求められたら作成モードの要領で直す。

## チェックリスト

- [ ] **網羅**：機能一覧(FN)に全機能が載っているか。実装/画面にあって FN に無い機能はないか。
- [ ] **トレーサビリティ**：各 FN に関連 SC/CM/CP/AC が埋まっているか。**存在しない ID への参照（リンク切れ）**は無いか。
- [ ] **重複行**：同名機能が複数行に分かれていないか（置き場所違いは関連画面列で表現すべき）。
- [ ] **受け入れ基準**：各仕様に AC があるか。**AC 空欄＝未設計**として指摘する。曖昧語（適切に/正しく）で逃げていないか。
- [ ] **状態網羅**：一覧/フォーム系で「0件・絞り込み0件・エラー・ローディング・権限なし・範囲外」が漏れていないか。
- [ ] **非機能**：性能/SEO/アクセシビリティ/対応環境/セキュリティが宣言され、AC に落ちているか。
- [ ] **共通化**：絞り込み/ページャ/META 等の機構が各ページに重複記述されていないか（CP に集約すべき）。
- [ ] **設計の原則違反**：別ドメインの同種機能を1行に潰していないか／META・ページNo付与が2か所の責務を混在させていないか。
- [ ] **客先体裁**：変更履歴・前提/用語・図（サイトマップ/遷移）があるか。専門用語に偏っていないか。
- [ ] **版・文書管理**：提出済みなら版・Gitタグが入っているか。機能一覧の文書管理表が最新か。

## 指摘の書き方（例）

> **[必須] SC-003 ニュース一覧／状態網羅**
> 「絞り込み結果0件」と「範囲外ページ（?page=999）」の状態が未記載。
> → 状態網羅表に2行追加し、それぞれ AC を採番（例 AC-021 0件表示＋解除導線／AC-026 範囲外は404）。

> **[推奨] 機能一覧 FN-012 ソート／受け入れ基準**
> AC 列が空欄＝未設計。`?sort=` の値ごとの並び順を AC 化する（AC-023）。
