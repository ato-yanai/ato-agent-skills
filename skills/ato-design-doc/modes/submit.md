# 提出モード（版上げ・タグ付け）

「客先に1つ以上の設計書を提出する」ときだけ実行する。**文書ごとに版とタグを進める**。
前提は SKILL.md の「方針」「共通規約（版管理）」。
**コミットメッセージの作法は [ato-git-commit-message](../../ato-git-commit-message/SKILL.md) に委ね、
タグ付けはこのスキルが持つ**（責務を分ける：コミットスキルはメッセージ専用でタグを知らない）。

## コミットとタグの関係（不変条件）

- **1タグ = コミットに貼るラベル**（コミットの境界ではない）。`<ファイル名>-NN` は「その設計書が
  **第N版に到達したコミット**」を指す。
- そのコミットには「版を第N版に上げた編集＋変更履歴の行（Gitタグ列＝`<ファイル名>-NN`）」が**含まれている**。
- **タグを打つ時点で、対象設計書はコミット済みで変更が残っていない**。これでタグの中身と提出物が一致する。
  （検査は対象ファイル単位。別文書が未コミットでも、対象が確定していれば打てる）
- **版番号＝タグ番号**（microcms 第2版 ⇔ `microcms-02`）。

## 複数文書を同時に提出するとき（per-doc タグ ≠ per-doc コミット）

タグはラベルなので、**コミットを文書数だけ分ける必要はない**。2通りどちらでもよい：

- **推奨：1提出 = 1コミット（変更した全文書をまとめて）＋ そのコミットに複数タグ。**
  例: 1つのコミットに `microcms-02` と `page-02` を両方付ける。最もシンプルで、検査も素直に通る。
- **文書ごとに別コミットにしたい場合：** 各文書をコミットしてから、その文書のタグを打つ。
  検査は `--file` の対象ファイルだけを見るので、他の文書が未コミットでも対象のタグは打てる。

## タグ規約

- タグは **`<ファイル名>-<版2桁>`**（例 `microcms-02`）。`<ファイル名>` は提出する設計書ファイルのベース名
  （`functions` / `site` / `microcms` / `page` / `common` 等）。
- 同じ提出で複数文書を出すなら、**同一コミットに複数タグ**を付ける（提出パッケージはそのコミットで束ねる）。

## 自動検査（タグ付け前のプリフライト）

スクリプトはタグを打つ前に次を検査し、ダメなら**打たずに中止**する（`--skip-checks` で省略可）。

1. **対象設計書（`--file`）がコミット済み・変更なし**（あれば「先にコミットして」と中止）。対象ファイル単位で見るので、別文書が未コミットでも止めない。
2. `--file` 指定時、**対象設計書に `第N版` と Gitタグ `<ファイル名>-NN` がある**（＝版が上がっている）。無ければ中止。
3. `--file` 指定時、**その版上げが HEAD のコミットに含まれているか**（対象ファイルを最後に変更したコミット＝HEAD か）。
   タグは HEAD に付くので、版上げが過去の別コミットにあると提出物とずれる。ずれていれば中止。
4. **HEAD が提出コミットらしいか**（`design` / `提出`）— 警告のみ。
5. タグ付け後、`タグ → コミット(hash・件名)` を表示して対応を確認できる。

## 手順（このモードが選ばれたら、この順で進める）

順序が保証の肝：**先に版・タグ情報を更新 → コミット → コミット後にタグ → 最後に push を促す**。
タグは「版が上がったコミット」に後付けするので、必ずコミット後・版上げ後に打つ。

1. **対象設計書を決める。** 提出する設計書（複数可）と、それぞれのファイルパスを確定する。
2. **次のタグ名（＝版）を確認する。** 設計書ごとに次番号を出す。これを版・タグ情報に正しく書くため。
   ```bash
   node scripts/tag-submission.mjs <ファイル名> --dry-run
   ```
3. **（任意）変更点を把握する。** 前の版のタグと作業ツリーの差分を見て要約する（生 diff は貼らない）。
   ```bash
   node scripts/tag-submission.mjs <ファイル名> --diff --file <設計書のパス>
   ```
4. **★まず版とタグ情報を更新する（最初のメイン作業）。** 変わった設計書だけ：
   - 版を +1（第N版。N は手順2で確認した番号）。
   - 変更履歴に1行追記（版・提出日・提出先・主な変更点・**Gitタグ＝`<ファイル名>-<版2桁>`**）。
   - 機能一覧の「文書管理（一式の版状況）」に現行版・最終提出日・最終提出タグを反映。
   - 変わっていない文書は据え置き。
5. **コミットする（コミットメッセージは委譲）。**
   - **`ato-git-commit-message` が使えるなら、それに従ってメッセージを作りコミットする。**
     提出コミットの印として **scope を `design` に固定**する（例 `chore(design): 設計書を第N版として提出`）。
   - **無ければ、ユーザーにコミットメッセージを入力してもらい、そのままコミットする**
     （その場合も `design` scope を勧める）。
6. **タグを付与する（コミット後）。** 提出した設計書ごとに実行（1コミットに複数タグ可）。`--file` 必須。
   ```bash
   node scripts/tag-submission.mjs <ファイル名> --file <設計書のパス>
   ```
   プリフライト検査（下記）を通ると、`<ファイル名>-NN` を HEAD（＝提出コミット）に打ち、タグ→コミットの対応を表示する。
   **タグ注釈は付けなくてよい**（既定で `<ファイル名>の第N版を提出` が入る）。`-m` で上書きもできるが、**客の社名は入れない**
   （提出先は変更履歴テーブルの「提出先・目的」列が持つ。タグ注釈は版の事実だけ）。**push はしない。**
7. **push を促して完了。** スクリプトが案内する push コマンドをユーザーに伝える（自動 push しない）。
   ```bash
   git push --follow-tags
   ```
   上げた版・打ったタグ・提出物を日本語でまとめて報告し、「最後に上記で push してください」と促す。

## git の順序メモ

- git は `commit` と `tag` が別操作。**タグは「既にあるコミット」に後付け**する（`git tag` は既定で HEAD に付く）。
  同時に作る単一コマンドは無く、`commit` → 直後に `tag` の順。よって「版を上げる → commit → tag」になる。
- **変更点（主な変更点）は commit に含めたいので、差分取得は commit より前。** タグ付けは commit 後。役割が前後に分かれる。

## なぜ「コミット時に自動タグ」にしないか

タグを打つべきは「提出コミット」だけで、日々の作業コミットには不要。毎コミットでの自動付与は誤り。
引き金は「提出」イベント（このフロー）であって、コミット一般ではない。

**完全自動化したい場合**（提出コミットを検知して自動でタグ）は、`chore(design): …提出` を目印に
`post-commit` / `pre-push` フックから `tag-submission.mjs` を呼ぶ。これは settings.json / git hook の
領分（update-config スキル）で、汎用のコミットメッセージスキルには入れない。
