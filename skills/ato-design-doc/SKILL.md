# Design Doc（設計書）

Webサイト/CMS構築の**客先提出設計書**を、PMが違っても同じ品質・同じ書き方で作るためのスキル。
「設計書を見たら開発からテストまで一気にできる」を目標に、**機能一覧を親に全設計をひも付け、
全仕様項目に受け入れ基準（テスト観点）を併記する**。

このスキルは**モード制**。起動したらまずモードを選び、選ばれたモードの手順ファイル（`modes/*.md`）に従う。
方針・共通規約はこの SKILL.md に集約し、全モードで共有する。

## 方針（これだけは守る）

- **結果・進捗・指摘の報告はすべて日本語で行う（絶対事項）。**
- **出力は Markdown。** テンプレは `assets/*.md`。コピーして埋める。表は崩さない。
- **機能一覧(FUNC)が親。** すべての設計（画面/データ/共通処理）は FUNC にひも付ける。FUNC のない仕様を作らない。
- **全仕様項目に受け入れ基準(AC)を併記する。** AC がそのままテストケースになる。AC 空欄は「未設計」として扱う。
- **能力で分け、仕様では分けない。** 置き場所・パラメータの違いは画面(PAGE)側に書く。能力が違うときだけ別 FUNC にする。
- **仕組みは共通処理(COMMON)に1つ、適用はドメインごとの FUNC に分離する。** 絞り込み・ページネーション・META生成等の
  機構は COMMON に集約し、各 FUNC はそれを参照する（DRY にするのは機構だけ）。
- **microCMS 設計は既存スキルと用語・型を揃える。** フィールド種別/型は
  [ato-microcms-types](../ato-microcms-types/SKILL.md)、取得条件は [ato-microcms-fetch](../ato-microcms-fetch/SKILL.md)、
  Webhook/再検証は [ato-microcms-webhook](../ato-microcms-webhook/SKILL.md) に合わせる。

## 起動時：モードを選ぶ（必ず最初に行う）

**このスキルが起動したら、何より先に AskUserQuestion でモードを尋ねる。** 勝手にモードを決めない。
ユーザーの最初の依頼文からモードが自明（例「設計書をレビューして」）でも、念のため確認して選ばせる。

提示する3モードと、選ばれたら読む手順ファイル：

| モード | 用途 | 手順ファイル |
| --- | --- | --- |
| **① 作成** | 新規に設計書を作る／追記する | `modes/create.md` |
| **② レビュー・監査** | 既存設計書の抜け漏れを検出して指摘する | `modes/review.md` |
| **③ 提出** | 客先提出に向け版を上げ、コミット＆タグ付けする | `modes/submit.md` |

選ばれたら、**該当の `modes/*.md` を読み、その手順に従う。** どのモードでも下の「共通規約」を前提にする。

## 設計の流れと前後関係（全モード共通）

本スキルが扱うのは「要件定義 → **設計** → 実装・テスト」の中段。前後は2本のIDの糸でつながる。
**FUNC が要件定義↔設計を、AC が設計↔テストをつなぐ**。だから設計の完了は「FUNC に全設計がぶら下がり、
全仕様に AC が書けたか」で機械的に判定できる。

```
要件定義 ──FUNC──▶ 設計（本スキル）──AC──▶ 実装・テスト
        「何を作るか」            「設計＝テストケース」
```

- **入口（要件定義から受け取る）：** 機能リスト＋優先度／サイトマップ案／コンテンツ項目＋運用／デザイン／非機能目標。
  揃っていない項目は空欄のまま進めず「客先確認事項」として残課題にする（受け取りチェックは `modes/create.md` の入力ゲート）。
- **出口（実装・テストへ渡せる条件）：** 全 FUNC に PAGE/CMS/COMMON/AC が埋まりリンク切れゼロ、かつ全仕様に AC がある
  ＝テストケースが導出できる状態（出口チェックは `modes/review.md` のチェックリスト）。

### 設計書を作る順序（依存関係）

5本はフラットではなく、上流が下流の前提になる。**①② を先に立て、③→④→⑤の順**。④と⑤は往復して確定する。

1. **機能一覧（親）** — FUNC を採番し NFR を宣言。トレーサビリティの起点。
2. **サイト全体（親）** — PAGE・URL・遷移を確定。ページ設計の単位(PAGE)がここで決まる。
3. **microCMS** — CMS・フィールド・**絞り込み/ソート軸**を確定。一覧仕様の前提。
4. **ページ設計** — PAGE 単位で要素・状態網羅・SEO・AC。②③が無いと書けない。
5. **共通処理** — ④で見えた横断機構を COMMON に集約（④を参照に書き換える）。

詳しい種別の選び方・手順は `modes/create.md`。

## 共通規約（全モード共通）

### 章立て標準（全設計書共通）

`# タイトル — 案件名` → `## 変更履歴`（版/提出日/提出先・目的/主な変更点/Gitタグ）→ 本文 →
必要なら `## 前提・制約` `## 用語集`。各設計書の冒頭で関連する親 ID（FUNC/PAGE）を示し、上から辿れるようにする。

### ID・トレーサビリティ規約

ID は **接頭辞（どの設計書のものか）＋ 連番** で付ける。接頭辞は設計書ファイルから連想できる語にし、
**ID を見ればどの設計書を見ればよいか分かる**ようにする。

| 接頭辞 | 対象 | 置き場所（ファイル） | 例 |
| --- | --- | --- | --- |
| `FUNC-` | 機能（能力） | 機能一覧 `functions.md` | `FUNC-010 ニュース一覧取得・表示` |
| `PAGE-` | 画面 | サイト全体 `site.md` ／ ページ設計書 `page-<slug>.md` | `PAGE-003 ニュース一覧` |
| `CMS-` | コンテンツモデル(API) | microCMS設計書 `microcms.md` | `CMS-002 news` |
| `COMMON-` | 共通処理（横断機構） | 共通処理設計書 `common.md` | `COMMON-005 一覧フィルタ機構` |
| `AC-` | 受け入れ基準 | 各仕様項目（業界慣用語のまま） | `AC-024 11件目以降は2ページ目に出る` |
| `NFR-` | 非機能要件 | 機能一覧 `functions.md` | `NFR-01 一覧LCP 2.5s以内` |

- **接頭辞＝どの設計書か。** ID を見たら接頭辞でどのファイルを開けばよいか分かる。`PAGE-` の実体ファイルは
  `site.md` の画面一覧で `page-<slug>.md` に対応づく（`PAGE-003` ⇄ `page-news.md`）。
- **採番はドメインで番号帯を分ける**（例 News=FUNC-0xx、Blog=FUNC-1xx）。連番ではなく桁で領域を表す。
- **FUNC↔PAGE は多対多。** TOPと一覧の両方に出す機能は 1 FUNC・関連画面列に複数 PAGE。**同名行を作らない。**
- **別ドメイン（別 CMS）への同種操作は別 FUNC。** ただし仕組みは COMMON に 1 つ置き、各 FUNC が参照する。
  例：ニュース絞り込み(FUNC) と ブログ絞り込み(FUNC) は別行で、ともに `COMMON-005 一覧フィルタ機構` を参照。
- **別行にする/しないの線引き**：置き場所・パラメータの違い → 同じ FUNC（差は PAGE 側に記述）。能力・ロジックが違う → 別 FUNC。

### 受け入れ基準(AC)の書き方

- 各仕様項目に**検証可能な形**で 1 つ以上書く。チェック可能な箇条書き、または Given/When/Then。
  - 例：`AC-024 /news?page=2 で 13〜24件目が表示される`／`AC-027 ?page=N で title 末尾に「(Nページ目)」が付く`
- **そのままテストケースになる粒度**にする（曖昧語「適切に」「正しく」を避け、入力と期待結果を書く）。
- 仕様変更時は、変えた仕様に紐づく AC を洗い出して更新する（トレーサビリティの目的）。

### 状態網羅の固定リスト（一覧・フォーム系で必ず潰す）

`正常 / 空（0件）/ 絞り込み0件 / エラー（取得失敗）/ ローディング / 権限なし / 範囲外（不正パラメータ）/ レスポンシブ(SP・PC)`

### 非機能の最低ライン（機能一覧の NFR で宣言）

`性能（LCP等）/ SEO（title・description・canonical・OGP・sitemap・noindex方針）/ アクセシビリティ（キーボード操作・代替テキスト）/ 対応環境（ブラウザ・端末）/ セキュリティ（XSS・入力検証・秘匿情報の扱い）`

### 版管理（履歴は2層・版は文書ごと独立）

履歴は2層で持つ。**日々の全変更は Git**（コミット/PR。[ato-git-commit-message](../ato-git-commit-message/SKILL.md) に従う）、
**変更履歴テーブルは「客先に提出した版」だけ**を記録する。日々の修正をテーブルに二重記載しない。

- **版は文書ごとに独立。** 提出時に、内容が変わった文書だけ版を整数 +1（第1版 → 第2版 …）。
  変わっていない文書は据え置く。提出時点で文書ごとに版が異なるのが正常。
- **提出前（初回提出まで）は版・行を作らない。** Git で進める。最初の提出で「第1版」の行を立てる。
- **提出のたびに、変わった文書ごとに Git タグを打つ。** タグは `<ファイル名>-<版2桁>`（例 `microcms-02` ＝第2版）。
  **タグ番号＝版番号。** 同じ提出で複数文書を出すなら同一コミットに複数タグを付ける（手順とスクリプトは `modes/submit.md`）。
  各変更履歴の `Gitタグ` 列で版とタグを対応させる。
- **主な変更点は前の版のタグとの差分から要約する。** `tag-submission.mjs <ファイル名> --diff --file <パス>` で
  差分を出し（commit 前）、それを要約して変更履歴に書く。
- **タグはコミット後に打ち、版が上がっているかを検査する。** スクリプトが「未コミット変更なし／対象に第N版・
  Gitタグ有り／その版上げが HEAD コミットに含まれる」を確認してから打つ（タグは HEAD に付くため）。
  コミットとタグの不変条件は `modes/submit.md` 参照。
- **一式の版状況は親（機能一覧）の「文書管理表」で一覧する。** 文書ごとに版がばらつくので、
  「今どの文書が何版か・最後にいつ出したか」を1か所で把握できるようにする。

変更履歴テーブルの列は `版 / 提出日 / 提出先・目的 / 主な変更点 / Gitタグ`。
「変更者」は Git が持つので客先向けには不要（必要なら提出者・承認者に置き換える）。

### ファイル分割と版の粒度

設計書が縦長になったら分割する。**分割の単位＝版の単位**にする（分割した各ファイルが独立した文書）。

- **page（ページ設計書）はページ単位に分割する。** 1 PAGE = 1ファイル `page-<slug>.md`。各ファイルが
  自前の変更履歴を持ち、**ページごとに版・タグを進める**（例 `page-news-02`）。画面の索引は
  `site.md` の画面一覧が兼ねる（page の索引ファイルは作らない）。
- **microcms.md / common.md は分割せず1ファイル。** それぞれ単一文書として版を進める（`microcms-NN` / `common-NN`）。
- **functions.md / site.md（親・索引）も1ファイル。**
- **slug 命名規約（ファイル名）：** ルートからファイル名を作る。セグメントを `-` 連結／先頭 `/` 除去／動的部 `[id]` は役割名
  （`detail` 等）／TOP は `top`。例：`/`→`page-top` ／ `/news`→`page-news` ／ `/news/[id]`→`page-news-detail` ／
  `/contact/complete`→`page-contact-complete` ／ 404→`page-404`。**トレーサビリティのキーは中の PAGE-ID**（ファイル名ではない）。
- **文書管理表（機能一覧）はページも1行ずつ載せる。** ページ単位で版が異なるので、「親・microcms・common」と
  「ページ設計書（ページ単位）」の2グループで一覧する。

## 設計の原則（全モード共通・つまずきやすい点）

作成時はこれに沿って書き、レビュー時はこれが守られているかを突く。

- **同名機能が2画面に出る**（TOPと一覧のニュース表示）→ 別行にしない。1 FUNC・関連画面列に複数 PAGE、件数等の差は各 PAGE に書く。
- **別ドメインの同種機能**（News絞り込み／Blog絞り込み、軸が違う）→ FUNC は別行・CMS も別。仕組みは `COMMON` を共有し、
  使える軸はページ設計書の「軸マトリクス」で画面ごとに示す。
- **ページが増えて行が爆発する**懸念 → 増えるのは**サイト全体の画面一覧(PAGE)**。機能一覧は「異なる能力の数」だけ増え、
  共通機能（一覧表示/詳細/フォーム/パンくず）は再利用で 1 行。マトリクスを機能一覧に展開しない。
- **META/ページNo付与が2か所に跨る** → 付与ロジック・命名規則は共通処理(COMMON)、このページで出す実際の文言・noindex方針は
  ページ設計書(PAGE)。混在させない。
- **relation/参照を絞り込み軸に使う** → microCMS 側で参照先フィールドと絞り込み可否を明記し、
  ato-microcms-types の relation の扱い（depth で形が変わる）に合わせる。
- **AC が書けない** → 仕様が未確定のサイン。空欄のまま提出せず「未確定」と明記し、客先確認事項として残課題に挙げる。
