# Design Doc（設計書）

Webサイト/CMS構築の**客先提出設計書**を、PMが違っても同じ品質・同じ書き方で作るためのスキル。
「設計書を見たら開発からテストまで一気にできる」を目標に、**機能一覧を親に全設計をひも付け、
全仕様項目に受け入れ基準（テスト観点）を併記する**。

このスキルは**モード制**。起動したらまずモードを選び、選ばれたモードの手順ファイル（`modes/*.md`）に従う。
方針・共通規約はこの SKILL.md に集約し、全モードで共有する。

## 方針（これだけは守る）

- **結果・進捗・指摘の報告はすべて日本語で行う（絶対事項）。**
- **出力は Markdown。** テンプレは `assets/*.md`。コピーして埋める。表は崩さない。
- **機能一覧(FN)が親。** すべての設計（画面/データ/共通処理）は FN にひも付ける。FN のない仕様を作らない。
- **全仕様項目に受け入れ基準(AC)を併記する。** AC がそのままテストケースになる。AC 空欄は「未設計」として扱う。
- **能力で分け、仕様では分けない。** 置き場所・パラメータの違いは画面(SC)側に書く。能力が違うときだけ別 FN にする。
- **仕組みは共通処理(CP)に1つ、適用はドメインごとの FN に分離する。** 絞り込み・ページネーション・META生成等の
  機構は CP に集約し、各 FN はそれを参照する（DRY にするのは機構だけ）。
- **microCMS 設計は既存スキルと用語・型を揃える。** フィールド種別/型は
  [ato-microcms-types](../ato-microcms-types/SKILL.md)、取得条件は [ato-microcms-fetch](../ato-microcms-fetch/SKILL.md)、
  Webhook/再検証は [ato-microcms-webhook](../ato-microcms-webhook/SKILL.md) に合わせる。

## 起動時：モードを選ぶ（必ず最初に行う）

**このスキルが起動したら、何より先に AskUserQuestion でモードを尋ねる。** 勝手にモードを決めない。
ユーザーの最初の依頼文からモードが自明（例「設計書をレビューして」）でも、念のため確認して選ばせる。

提示する3モードと、選ばれたら読む手順ファイル：

| モード | 用途 | 手順ファイル |
| --- | --- | --- |
| **① 作成** | 新規に設計書を作る／追記する | `modes/create.md` |
| **② レビュー・監査** | 既存設計書の抜け漏れを検出して指摘する | `modes/review.md` |
| **③ 提出** | 客先提出に向け版を上げ、コミット＆タグ付けする | `modes/submit.md` |

選ばれたら、**該当の `modes/*.md` を読み、その手順に従う。** どのモードでも下の「共通規約」を前提にする。

## 構成

| 種別 | ファイル | 役割 |
| --- | --- | --- |
| モード手順 | `modes/create.md` | 作成モードの手順・種別の選び方・例 |
| モード手順 | `modes/review.md` | レビュー/監査モードのチェックリスト・指摘の書き方 |
| モード手順 | `modes/submit.md` | 提出フロー（版上げ・タグ付け） |

## 共通規約（全モード共通）

### 章立て標準（全設計書共通）

`# タイトル — 案件名` → `## 変更履歴`（版/提出日/提出先・目的/主な変更点/Gitタグ）→ 本文 →
必要なら `## 前提・制約` `## 用語集`。各設計書の冒頭で関連する親 ID（FN/SC）を示し、上から辿れるようにする。

### ID・トレーサビリティ規約

| 接頭辞 | 対象 | 置き場所 | 例 |
| --- | --- | --- | --- |
| `FN-` | 機能（能力） | 機能一覧 | `FN-010 ニュース一覧取得・表示` |
| `SC-` | 画面 | サイト全体／ページ設計書 | `SC-003 ニュース一覧` |
| `CM-` | コンテンツモデル(API) | microCMS設計書 | `CM-002 news` |
| `CP-` | 共通処理（横断機構） | 共通処理設計書 | `CP-005 一覧フィルタ機構` |
| `AC-` | 受け入れ基準 | 各仕様項目 | `AC-024 11件目以降は2ページ目に出る` |
| `NFR-` | 非機能要件 | 機能一覧 | `NFR-01 一覧LCP 2.5s以内` |

- **採番はドメインで番号帯を分ける**（例 News=FN-0xx、Blog=FN-1xx）。連番ではなく桁で領域を表す。
- **FN↔SC は多対多。** TOPと一覧の両方に出す機能は 1 FN・関連画面列に複数 SC。**同名行を作らない。**
- **別ドメイン（別 CM）への同種操作は別 FN。** ただし仕組みは CP に 1 つ置き、各 FN が参照する。
  例：ニュース絞り込み(FN) と ブログ絞り込み(FN) は別行で、ともに `CP-005 一覧フィルタ機構` を参照。
- **別行にする/しないの線引き**：置き場所・パラメータの違い → 同じ FN（差は SC 側に記述）。
  能力・ロジックが違う → 別 FN。

### 受け入れ基準(AC)の書き方

- 各仕様項目に**検証可能な形**で 1 つ以上書く。チェック可能な箇条書き、または Given/When/Then。
  - 例：`AC-024 /news?page=2 で 13〜24件目が表示される`／`AC-027 ?page=N で title 末尾に「(Nページ目)」が付く`
- **そのままテストケースになる粒度**にする（曖昧語「適切に」「正しく」を避け、入力と期待結果を書く）。
- 仕様変更時は、変えた仕様に紐づく AC を洗い出して更新する（トレーサビリティの目的）。

### 状態網羅の固定リスト（一覧・フォーム系で必ず潰す）

`正常 / 空（0件）/ 絞り込み0件 / エラー（取得失敗）/ ローディング / 権限なし / 範囲外（不正パラメータ）/ レスポンシブ(SP・PC)`

### 非機能の最低ライン（機能一覧の NFR で宣言）

`性能（LCP等）/ SEO（title・description・canonical・OGP・sitemap・noindex方針）/ アクセシビリティ（キーボード操作・代替テキスト）/ 対応環境（ブラウザ・端末）/ セキュリティ（XSS・入力検証・秘匿情報の扱い）`

### 版管理（履歴は2層・版は文書ごと独立）

履歴は2層で持つ。**日々の全変更は Git**（コミット/PR。[ato-git-commit-message](../ato-git-commit-message/SKILL.md) に従う）、
**変更履歴テーブルは「客先に提出した版」だけ**を記録する。日々の修正をテーブルに二重記載しない。

- **版は文書ごとに独立。** 提出時に、内容が変わった文書だけ版を整数 +1（第1版 → 第2版 …）。
  変わっていない文書は据え置く。提出時点で文書ごとに版が異なるのが正常。
- **提出前（初回提出まで）は版・行を作らない。** Git で進める。最初の提出で「第1版」の行を立てる。
- **提出のたびに、変わった文書ごとに Git タグを打つ。** タグは `<ファイル名>-<版2桁>`（例 `microcms-02` ＝第2版）。
  **タグ番号＝版番号。** 同じ提出で複数文書を出すなら同一コミットに複数タグを付ける（手順とスクリプトは `modes/submit.md`）。
  各変更履歴の `Gitタグ` 列で版とタグを対応させる。
- **主な変更点は前の版のタグとの差分から要約する。** `tag-submission.mjs <ファイル名> --diff --file <パス>` で
  差分を出し（commit 前）、それを要約して変更履歴に書く。
- **タグはコミット後に打ち、版が上がっているかを検査する。** スクリプトが「未コミット変更なし／対象に第N版・
  Gitタグ有り／その版上げが HEAD コミットに含まれる」を確認してから打つ（タグは HEAD に付くため）。
  コミットとタグの不変条件は `modes/submit.md` 参照。
- **一式の版状況は親（機能一覧）の「文書管理表」で一覧する。** 文書ごとに版がばらつくので、
  「今どの文書が何版か・最後にいつ出したか」を1か所で把握できるようにする。

変更履歴テーブルの列は `版 / 提出日 / 提出先・目的 / 主な変更点 / Gitタグ`。
「変更者」は Git が持つので客先向けには不要（必要なら提出者・承認者に置き換える）。

## 設計の原則（全モード共通・つまずきやすい点）

作成時はこれに沿って書き、レビュー時はこれが守られているかを突く。

- **同名機能が2画面に出る**（TOPと一覧のニュース表示）→ 別行にしない。1 FN・関連画面列に複数 SC、件数等の差は各 SC に書く。
- **別ドメインの同種機能**（News絞り込み／Blog絞り込み、軸が違う）→ FN は別行・CM も別。仕組みは `CP` を共有し、
  使える軸はページ設計書の「軸マトリクス」で画面ごとに示す。
- **ページが増えて行が爆発する**懸念 → 増えるのは**サイト全体の画面一覧(SC)**。機能一覧は「異なる能力の数」だけ増え、
  共通機能（一覧表示/詳細/フォーム/パンくず）は再利用で 1 行。マトリクスを機能一覧に展開しない。
- **META/ページNo付与が2か所に跨る** → 付与ロジック・命名規則は共通処理(CP)、このページで出す実際の文言・noindex方針は
  ページ設計書(SC)。混在させない。
- **relation/参照を絞り込み軸に使う** → microCMS 側で参照先フィールドと絞り込み可否を明記し、
  ato-microcms-types の relation の扱い（depth で形が変わる）に合わせる。
- **AC が書けない** → 仕様が未確定のサイン。空欄のまま提出せず「未確定」と明記し、客先確認事項として残課題に挙げる。
