# ato-design-doc

Webサイト/CMS構築の**客先提出設計書**を、担当PMが違っても同じ品質・同じ書き方で作るためのスキルです。

> このファイルは**人間向けの概要**です。エージェントへの指示の正（Single Source of Truth）は
> [`SKILL.md`](SKILL.md)。モード別の手順は [`modes/`](modes/)、テンプレは [`assets/`](assets/) にあります。

---

## 何をするスキルか

「設計書を見たら、開発からテストまで一気にできる」を目標に、次の2点を強制します。

- **機能一覧(FUNC)を親に、すべての設計（画面/データ/共通処理）をひも付ける。** FUNC のない仕様を作らない。
- **全仕様項目に受け入れ基準(AC)を併記する。** AC がそのままテストケースになる。AC 空欄＝「未設計」。

この2本の糸で、**要件定義 ↔ 設計**（FUNC）と **設計 ↔ テスト**（AC）のトレーサビリティを取り、
状態網羅・非機能・版管理の抜けを構造的に防ぎます。

```
要件定義 ───── FUNC ─────▶ 設計（本スキル）───── AC ─────▶ 実装・テスト
            「何を作るか」                 「設計＝テストケース」
```

## 設計書5本（構成）

| 種別 | ファイル | 役割 | 主なID |
| --- | --- | --- | --- |
| 機能一覧（親） | [`assets/functions.md`](assets/functions.md) | 機能の一覧・非機能要件・文書管理表。トレーサビリティの起点 | `FUNC-` / `NFR-` |
| サイト全体設計（親） | [`assets/site.md`](assets/site.md) | サイトマップ・画面一覧・遷移・URL設計 | `PAGE-` |
| microCMS設計書 | [`assets/microcms.md`](assets/microcms.md) | コンテンツモデル・フィールド・絞り込み/ソート軸・権限・Webhook（具体的な取得条件はページ設計書） | `CMS-` |
| ページ設計書 | [`assets/page.md`](assets/page.md) → `page-<slug>.md` | **1ページ＝1ファイル**。画面ごとの要素・状態網羅・SEO・受け入れ基準 | `PAGE-` / `AC-` |
| 共通処理設計書 | [`assets/common.md`](assets/common.md) | 絞り込み/ページャ/META/フォーム/エラー等の横断機構 | `COMMON-` / `AC-` |

> **IDの接頭辞＝どの設計書か。** `FUNC-`＝機能一覧、`PAGE-`＝画面、`CMS-`＝microCMS、`COMMON-`＝共通処理。
> IDを見れば開くべき設計書が分かる（数字は連番、ドメインで番号帯を分ける）。`AC-`（受入基準）/`NFR-`（非機能）は業界慣用語のまま。

> **依存関係:** ① 機能一覧 と ② サイト全体 を先に立て、③ microCMS → ④ ページ設計 → ⑤ 共通処理 の順に埋める。
> ④と⑤は往復しながら確定する（横断機構はページから COMMON に吸い上げる）。詳しくは [`modes/create.md`](modes/create.md)。

> **ファイル分割と版:** ページ設計書だけ **1ページ＝1ファイル `page-<slug>.md`** に分け、**ページごとに版・タグを独立**で進める
> （例 `page-news-02`）。機能一覧・サイト全体・microCMS・共通処理は各1ファイルで各自の版を持つ。詳しくは SKILL.md「ファイル分割と版の粒度」。

## 3つのモード

起動したらまず **AskUserQuestion でモードを選ぶ**（勝手に決めない）。選ばれたモードの手順ファイルに従います。

### ① 作成 — `modes/create.md`

新規に設計書を作る／既存に追記するモード。

- 新規CMS案件のデフォルトは、まず**親2本（機能一覧・サイト全体）**を立て、そこから各設計書を埋める。
- `assets/*.md` のテンプレをコピーし、案件名・日付・内容を埋める。出力先は `docs/design/<種別>.md` を提案。
- SKILL.md「ID・トレーサビリティ規約」に従い **FUNC/PAGE/CMS/COMMON/AC** を採番。FUNC↔PAGE は多対多で、同名行を作らない。
- 各仕様に **AC を必ず採番**し、一覧/フォーム系は「状態網羅の固定リスト」を必ず潰す。
- 新しい FUNC/画面は親（機能一覧・サイト全体）へ反映し、参照IDがすべて解決すること（リンク切れゼロ）を確認。

### ② レビュー・監査 — `modes/review.md`

既存の設計書の抜け漏れを検出して指摘するモード。**「実装・テストへ渡せるか」を判定する出口ゲート**（① 作成の入力ゲートと対になる）。

- チェックリストが go/no-go の合否表。**A. トレーサビリティ（実装に渡せるか）／B. テスト基盤（AC・状態網羅・非機能）／C. データ・画面の確定／D. 客先体裁・版／E. 残課題**の5区分で突く。
- 各指摘は **「箇所（ID/章）→ 何が欠けているか → どう直すか」** の順で、重大度（必須修正/推奨/軽微）を付けて日本語で書く。
- 最後に **go/no-go を明言**する（必須修正が残っていれば no-go）。AC 空欄はテストケースを導出できないため必須修正扱い。

### ③ 提出 — `modes/submit.md`

客先に1つ以上の設計書を提出するときだけ実行する。**文書ごとに版とタグを進める**モード。

- **責務分担:** コミットメッセージの作法は [ato-git-commit-message](../ato-git-commit-message/SKILL.md) に委ね、
  **タグ付けはこのスキルが持つ**。
- **版・タグ規約:** 版は文書ごとに独立。提出時に内容が変わった文書だけ版を整数+1。
  タグは **`<ファイル名>-<版2桁>`**（例 `microcms-02` ＝第2版）で、**タグ番号＝版番号**。
- **不変条件:** 1タグ＝コミットに貼るラベル。`<ファイル名>-NN` は「その設計書が第N版に到達したコミット」を指し、
  タグを打つ時点で対象設計書はコミット済み・変更なし。これで提出物とタグの中身が一致する。
- **順序が保証の肝:** `版・タグ情報を更新 → コミット → コミット後にタグ → 最後に push を促す`。
  タグは「版が上がったコミット」に後付けするため、必ずコミット後・版上げ後に打つ。
- **自動検査:** [`scripts/tag-submission.mjs`](scripts/tag-submission.mjs) がタグ付け前にプリフライト検査を行い、
  通らなければ**打たずに中止**する。push は自動で行わず、コマンドを促すだけ。

主なコマンド（依存ゼロ。詳細は [`modes/submit.md`](modes/submit.md)）:

```bash
node scripts/tag-submission.mjs <ファイル名> --dry-run                 # 次のタグ名（=版）を確認
node scripts/tag-submission.mjs <ファイル名> --diff --file <パス>      # 前タグとの差分（commit前・要約用）
node scripts/tag-submission.mjs <ファイル名> --file <パス>             # コミット後にタグ付け＋対応表示
```

> **複数文書を同時提出**するときは、1コミットに変更した全文書をまとめ、そのコミットに複数タグを付けるのが推奨。
> 完全自動化（提出コミット検知でタグ）は git hook の領分で、`update-config` スキルで扱う。

## 使い方

1. このスキルを起動する（設計書の作成・レビュー・提出を依頼する）。
2. 提示される3モードから選ぶ。
3. 選んだモードの手順に従って進める。**結果・進捗・指摘の報告はすべて日本語**。
